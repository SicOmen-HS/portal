## Background

Projektet saknar en uttrycklig och operativ regel om att repositoryarbete normalt ska ske i en separat branch per arbetsobjekt eller avgränsat arbetsmoment.

Färdiga promptar och handoffs har vid upprepade tillfällen delats upp mellan flera text- och kodblock, vilket gör dem svåra att kopiera och använda i andra implementationagenter.

## Goal

Inför en enhetlig Git- och branchprincip samt ett bindande krav på sammanhängande, självständiga och lättkopierade leveranser.

## Scope

- Uppdatera docs/project/PROJECT_RULES.md med bindande regler för branchbaserat arbete och kopierbara leveranser.
- Uppdatera docs/project/PROJECT_WORKFLOW.md med en säker operativ branchrutin och krav för task prompts och handoffs.
- Uppdatera docs/project/DOCUMENT_INDEX.md endast om dess beskrivningar, Situation Guide eller katalog behöver ändras.
- Tillåt metadata- och köändringar som Worksmith själv utför.

## Requirements

- Aktiv utveckling ska normalt ske i en separat, liten och kortlivad branch per Worksmith-arbetsobjekt eller tydligt avgränsat och granskningsbart arbetsmoment, inte direkt på main; branchens namn ska innehålla arbetsobjektets id när ett sådant finns och branchen ska inte vara en permanent personlig utvecklingsbranch.
- Förändringar ska normalt levereras genom pull request och granskas före merge; main ska representera den senaste godkända och verifierade versionen och branchen ska tas bort efter godkänd merge.
- Commit, push, rebase, merge, force-push och andra Git-operationer ska förbli manuella och ägarstyrda; Worksmith utför eller godkänner dem inte.
- En färdig prompt, handoff, manifesttext, instruktion eller sammanhängande kommandosekvens ska levereras som ett enda komplett, självständigt och lättkopierat block med all nödvändig kontext.
- Förklaringar ska placeras före eller efter blocket, aldrig mitt i det, och instruktioner i samma leverans får inte spridas mellan kommentarer eller flera block enbart för läsbarhet.
- Regeln för kopierbara leveranser ska uttryckligen omfatta uppdrag till Codex, Claude Code och andra implementationagenter.
- Den operativa branchrutinen ska använda säkra PowerShell-kommandon, kontrollera arbetskatalogen före mutation och täcka uppdatering, branchskapande, verifiering, synkronisering mot origin/main, konflikthantering, push, pull request, granskning och borttagning efter merge.

## Verification

- Kör git diff --check.
- Kör npm.cmd run project -- validate.
- Kontrollera att nya eller ändrade relativa Markdown-länkar pekar på befintliga filer och att inga befintliga länkar brutits.
- Kontrollera att endast avsedda filer har ändrats och att reglerna inte duplicerar eller motsäger befintlig styrning.

## Acceptance Criteria

- PROJECT_RULES.md innehåller de bindande branchreglerna och regeln för ett enda komplett kopierbart leveransblock.
- PROJECT_WORKFLOW.md innehåller en säker PowerShell-baserad branchrutin samt motsvarande krav för task prompts och handoffs.
- DOCUMENT_INDEX.md ändras endast om discoverability faktiskt påverkas.
- Worksmith-validering och git diff --check är gröna och endast avsedda dokumentations- och Worksmith-filer har ändrats.

## Out of Scope

- Ändringar i Worksmiths programkod, schema, statusmodell eller CLI-beteende.
- Ändringar i applikationskod eller mockdata.
- Repositoryinställningar hos Git-leverantören.
- Commit, push, merge eller deploy.
- Nya paket eller uppgraderade Node-beroenden.

## Deliverables

- Uppdaterade styr- och workflowdokument enligt godkänt scope.
- Worksmith-arbetsobjekt och kö synkroniserade via CLI.
- Verifierad handoff som ett enda komplett och lättkopierat block.
