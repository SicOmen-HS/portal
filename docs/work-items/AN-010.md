## Background

AB-027 (slutförd, mergad) etablerade en lokal .NET Web API-backend (backend/Portal.Api) som bevisar kedjan Angular -> lokalt .NET Web API -> SqlServerDatasetSourceAdapter -> lokal SQL Server, dokumenterad i backend/README.md.

Fem senare commits (421d309, 00de6ce, fa5be78, a3bbda8, 9f4a2b1), skapade utanför denna sessions Worksmith-flöde, lade till en parallell Node.js/TypeScript-backend-POC (backend/package.json, backend/tsconfig.json, backend/src/server.ts, backend/src/trino.ts) i samma backend/-katalog som Portal.Api, tillsammans med två nya dokument i repo-roten (docs/LAB_LAKEHOUSE_PLAN.md, docs/PORTAL_LAKEHOUSE_INTEGRATION.md) och en samordningsnotis (docs/coordination/portal-lakehouse-frontend-coordination.md) om ett labb för Trino, Lakekeeper, Iceberg och MinIO/S3. backend/README.md, som tidigare dokumenterade AB-027:s .NET-POC, ersattes med innehåll om den nya Node-backenden istället.

Dessa ändringar är inte klassificerade mot projektets dokumenterade målarkitektur (Angular -> .NET Web API -> integrationsadaptrar, docs/04_Systemarkitektur.md) eller mot docs/07_AI_Instruktioner.md och docs/project/PROJECT_RULES.md, och har inte gått genom Worksmith-arbetsflödet. Ett litet, avgränsat analysarbete behövs för att klassificera nuläget och ta fram ett beslutsunderlag innan eventuell korrigering eller arkitekturbeslut.

## Goal

Ta fram ett neutralt, evidensbaserat beslutsunderlag om vad den nuvarande Node/TypeScript/Trino-POC:n faktiskt bevisar, hur den förhåller sig till den dokumenterade målarkitekturen och till AB-027:s .NET-backend, hur backend/README.md bör återställas, samt vilka små, avgränsade AB-item eller ADR-arbeten som bör följa. Analysen ska besvara: (1) vad POC:n faktiskt bevisar, (2) om Node-servern är tydligt avgränsad som ett tillfälligt lokalt labb eller framstår som en alternativ portalbackend, (3) hur lösningen förhåller sig till målbilden Angular -> .NET Web API -> integrationsadaptrar, (4) om en framtida Trino-integration bör kapslas bakom en adapter i den befintliga .NET-backenden, (5) om Node-POC:n bör ligga kvar under backend/, flyttas till ett tydligt labbområde eller senare tas bort, (6) hur dokumentationen för AB-027 och SQL Server-preview-POC:n ska återställas efter att backend/README.md ersattes, (7) om det finns säkerhets-, konfigurations-, underhålls- eller strukturproblem som bör hanteras i efterföljande arbete, och (8) vilka små och tydligt avgränsade AB:n eller eventuella ADR-arbeten bör följa.

## Korrigering efter granskning (Changes Requested)

Projektägarens granskning returnerade **Changes Requested** med åtta
begärda korrigeringar. Itemet flyttades från `needs_review` tillbaka till
`in_progress` via `npm.cmd run project -- transition AN-010 needs_review
in_progress` (dry-run kört före faktisk transition), korrigeringarna
gjordes enbart i analysrapporten och denna work item-fil, och itemet
flyttades därefter tillbaka till `needs_review`.

Korrigeringarna:

1. **Git-spårbarhet:** all formulering om att commitsen "skapades direkt på
   main" togs bort. Ersatt genomgående (bakgrund, analys, slutsats, work
   item) med: commitsen ligger linjärt på `main`/`origin/main` efter
   AB-027; ingen separat Worksmith-spårbarhet hittades; repositoryhistoriken
   kan inte ensam avgöra om de kom genom direkt push, en rebase-mergad
   pull request eller ett annat Git-flöde; ingen slutsats dras om att
   direkt push skett.
2. **Personuppgifter borttagna:** commitförfattarens namn och e-postadress
   togs bort från rapporten och work item-filen. Ersatt med
   "commitförfattaren"/"en parallellt arbetande kollega". Commit-id:n
   behölls för teknisk spårbarhet.
3. **Trino/lineage nyanserat:** "fel verktyg för lineage-relationer" togs
   bort. Ersatt med att direkta datafrågor via Trino inte i sig skapar
   eller bevisar lineage, att Trino möjligen kan bidra med teknisk metadata
   eller query-/eventinformation till ett metadatasystem (ej implementerat
   eller verifierat i denna POC), och att full lineage kräver modellerade
   relationer och en auktoritativ metadata-/lineagekälla.
4. **Dataåtkomst separerad från lineage/metadata:** rekommendationen
   (fråga 8, "Rekommenderad backendriktning") skriver nu uttryckligen att
   en framtida Trino-adapter för dataåtkomst endast är relevant mot ett
   identifierat portalbehov, att lineage bör konsumeras genom en adapter
   mot en auktoritativ metadata-/lineagekälla (t.ex. OpenMetadata, om
   bekräftat) snarare än en direkt `LakekeeperAdapter`, och att en adapter
   nämnd som exempel i docs/04 inte i sig innebär att den ska implementeras.
5. **Ny öppen fråga:** om full lineagevisualisering i portalen är ett
   förankrat produktbehov, en framtida hypotes eller en individuell teknisk
   vision, och i så fall vilken visningsform (full graf, sammanfattning
   eller länk till OpenMetadata). Det föreslagna framtida AN:et om
   lineage-modellering är nu uttryckligen villkorat av att denna fråga
   besvaras först.
6. **Neutralare språk:** "konkurrerar med" ersatt med "skapar strukturell
   och dokumentationsmässig oklarhet om vilken backend som är portalens
   ordinarie backend". "Förlorade" (om AB-027:s instruktioner) ersatt med
   "finns inte längre som aktiv dokumentation i nuvarande repositoryversion
   (kan återvinnas från git-historiken)".
7. **Miljöspecifika värden omklassificerade:** det specifika värdnamnet och
   sökvägarna behandlas nu som en generisk repository-hygienfråga att lösa
   i ett efterföljande AB, utan att upprepa de exakta värdena flera gånger
   eller ta ställning till vem miljön tillhör. Ingen befintlig
   lakehouse-dokumentation ändrades.
8. **Verifieringsredovisning förtydligad:** rapporten anger nu uttryckligen
   att `git diff --stat` endast redovisar redan spårade ändringar
   (`docs/WORK_QUEUE.md`) och att de två nya, untracked filerna verifierades
   separat via `git status --short --branch` och innehållsgranskning — en
   begränsning i kommandots redovisning, inte ett verifieringsfel.
