## Background

AB-027 (slutförd, mergad) skapade en lokal .NET-baserad backend-POC (backend/Portal.Api) med riktningen Angular -> .NET Web API -> adapter -> lokal SQL Server-preview, ursprungligen dokumenterad i backend/README.md.

Efter AB-027 tillkom en separat Node.js/TypeScript/Trino-baserad lakehouse-POC. Vid AN-010:s granskning hittades inget separat Worksmith-item kopplat till lakehouse-arbetet. Den skrev över backend/README.md så att filen idag uteslutande beskriver Node/Trino-labbet, medan .NET-POC:n under backend/Portal.Api saknar egen aktiv README.

AN-010 (slutförd, status done, validerad med 0 fel och 0 varningar, mergad till main via PR #13) har klassificerat Node/Trino-lösningen som ett tillfälligt tekniskt labb och en smal dataåtkomst-POC - inte portalens ordinarie backend och inte en verifiering av metadata eller lineage. Portalens dokumenterade ordinarie backendriktning kvarstår: Angular -> .NET Web API -> integrationsadaptrar.

Detta AB ska separera dokumentationen för de två POC:erna utan att ändra implementation, arkitektur eller repositorystruktur.

## Goal

Separera dokumentationen för portalens .NET-baserade backend-POC från den tillfälliga Node.js/TypeScript/Trino-baserade lakehouse-POC:n. Varje POC ska få en egen tydlig dokumentationsyta, och backendkatalogens README ska bli en neutral översikt som inte presenterar lakehouse-labbet som hela portalens backend.

## Scope

- Göra backend/README.md till en kort och neutral översikt över backendkatalogen.
- Tydliggöra i backend/README.md att .NET Web API är portalens dokumenterade ordinarie backendriktning.
- Länka från backend/README.md till dokumentationen för .NET-POC:n och till dokumentationen för lakehouse-labbet.
- Skapa backend/Portal.Api/README.md för den nuvarande .NET-POC:n från AB-027, med projektstruktur, förutsättningar, lokal SQL Server-preview, hur den lokala POC-databasen skapas, user-secrets utan riktiga credentials, körning, verifiering, återställning och relevanta endpoints.
- Använda den tidigare versionen av backend/README.md (commit 134f2b4) och AB-027 som källor, men verifiera varje kommando, filnamn, endpoint, projektsökväg och konfigurationsinstruktion mot nuvarande repository innan dokumentationen skrivs.
- Skapa backend/LAKEHOUSE_POC.md för Node.js/TypeScript/Trino-labbet, och flytta dit de befintliga tekniska instruktionerna från backend/README.md.
- Märka lakehouse-lösningen tydligt som ett tillfälligt tekniskt labb, en dataåtkomst-POC, inte portalens ordinarie backend, och inte en verifiering av metadata eller lineage.
- Hänvisa till AN-009, AB-027 och AN-010 där det är relevant.
- Behålla den tekniska innebörden i befintliga instruktioner.
- Ersätta interna eller miljöspecifika värden i de dokumentationsfiler som ändras med generiska exempel eller platshållare, förutom PortalPocLocal (se Requirements).

## Requirements

- PortalPocLocal ska behållas som namnet på .NET-POC:ns lokala SQL Server-databas i dokumentationen - det är ett generiskt, repository-lämpligt POC-namn för en lokal databas med en liten fiktiv datamängd, inte en delad testmiljö, acceptansmiljö, produktionsmiljö eller portalens framtida applikationsdatabas. Ersätt det inte med en platshållare.
- Dokumentationen ska tydliggöra att PostgreSQL fortfarande är portalens beslutade applikationsdatabas.
- Riktiga SQL Server-servernamn, instansnamn, användarnamn, lösenord, credentials och fullständiga miljöspecifika connection strings får inte dokumenteras.
- Repositoryt får inte tillföras eller återinföra interna URL:er, interna servernamn, interna värdnamn, absoluta användarsökvägar, absoluta server- eller labbsökvägar, secrets, lösenord, tokens, certifikat, riktiga connection strings, AD-grupper, produktionsdata, personuppgifter eller interna miljönamn. Använd generiska platshållare (t.ex. <trino-host>, <trino-port>, <path-to-lakehouse-lab>, <username>, <password>, <connection-string>) där värdet är miljöberoende eller potentiellt internt.
- Ofarliga och avsiktligt generiska lokala POC-värden (t.ex. PortalPocLocal, http://localhost-adresser som redan används i övrig projektdokumentation) får behållas när de gör instruktionerna körbara.
- Angular är frontend, .NET Web API är portalens dokumenterade ordinarie backendriktning, PostgreSQL är portalens applikationsdatabas. Frontend får inte prata direkt med interna system eller databaser.
- Node/Trino-POC:n ska inte beskrivas som portalens ordinarie backend. Trino och dataåtkomst får inte blandas ihop med metadata eller lineage. Full lineagevisning är inte ett redan bekräftat produktbehov.
- Skapa inga parallella begrepp och föreslå inga nya lösningar som inte krävs för dokumentationsarbetet.
- Ingen kod-, projekt-, beroende-, konfigurations- eller SQL-fil ändras. Endast dokumentationsfiler och nödvändiga Worksmith-filer ändras.
- Kod, tekniska identifierare och filnamn på engelska; dokumentation och användarnära text på svenska.
- Ingen commit, push, rebase, merge eller deploy.

## Acceptance Criteria

- backend/README.md är en kort och neutral backendöversikt som skiljer tydligt mellan portalens dokumenterade .NET-riktning och det tillfälliga lakehouse-labbet, länkar till backend/Portal.Api/README.md och backend/LAKEHOUSE_POC.md, och innehåller inga detaljerade körinstruktioner för respektive POC.
- backend/Portal.Api/README.md dokumenterar den nuvarande .NET-POC:n, verifierad mot nuvarande repository, beskriver PortalPocLocal som en lokal POC-databas med en liten fiktiv datamängd, tydliggör att PortalPocLocal inte är portalens framtida applikationsdatabas, och att PostgreSQL fortfarande är portalens beslutade applikationsdatabas.
- backend/LAKEHOUSE_POC.md dokumenterar Node.js/TypeScript/Trino-labbet och markerar det som ett tillfälligt tekniskt labb och en dataåtkomst-POC, inte portalens ordinarie backend, och inte en verifiering av metadata eller lineage.
- Inga tekniska instruktioner går förlorade; samtliga är verifierade mot nuvarande repository.
- Dokumentationen innehåller inga interna URL:er, servernamn, absoluta person- eller miljöspecifika sökvägar, secrets, certifikat, riktiga connection strings, AD-grupper, produktionsdata eller personuppgifter.
- Ingen kod-, projekt-, beroende-, konfigurations- eller SQL-fil ändras; endast dokumentationsfiler och nödvändiga Worksmith-filer ändras.
- npm.cmd run project -- validate är godkänd.
- git diff --check är godkänd.
