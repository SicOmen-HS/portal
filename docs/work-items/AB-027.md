## Background

AN-009 avgränsade en lokal backend-POC för Datamarknad: Angular -> .NET Web API -> metadataadapter -> lokal SQL Server med helt fiktiv data. AN-009 är godkänt, mergat och slutfört. Kedjan har hittills bara analyserats på konceptnivå; ingen kod, inget SQL-script och ingen backend har skapats.

Projektägaren vill nu bevisa kedjan praktiskt: att portalen kan läsa faktiska datarader från en lokal SQL Server-tabell via en riktig, begränsad SELECT, och visa dessa helt fiktiva rader i den befintliga previewytan (DatasetFieldsPreviewComponent, AB-023), utan att frontend någonsin ansluter direkt till SQL Server. Lokalt är SQL Server 2025 Developer, SQL Server Management Studio och .NET 10 SDK installerade av projektägaren; inga anslutningsvärden ska skrivas in i repositoryt.

## Goal

Bevisa en fullständig, lokal vertikal kedja: Angular -> lokalt .NET Web API -> SqlServerDatasetSourceAdapter -> lokal SQL Server -> en riktig, radbegränsad SELECT -> helt fiktiva datarader visas i portalens befintliga preview, för exakt en fiktiv datamängd (dataset-sales-transactions-demo), utan att bryta portalens versionshanterade mockläge och utan att frontend ansluter direkt till SQL Server.

## Scope

- Skapa ett minimalt .NET Web API-projekt (backend/Portal.Api, target framework net10.0) med GET /health, GET /api/datasets/{id} och GET /api/datasets/{id}/preview.
- Skapa backendägda DTO:er: DatasetDetailDto, DatasetFieldDto, DatasetPreviewDto.
- Skapa IDatasetSourceAdapter och SqlServerDatasetSourceAdapter för previewrader, med ett enda backendägt register (t.ex. KnownDatasetsRegistry) som ensam källa för känt dataset-id, statisk metadata och tillåten SQL-tabell/vy med kolumnmappning. Inga dubblerade eller parallella allowlists.
- GET /api/datasets/{id} använder statisk, registerbaserad metadata (ingen SQL-läsning). GET /api/datasets/{id}/preview bevisar den riktiga, parameteriserade, radbegränsade (TOP 10) SQL Server-läsningen mot exakt en fiktiv, typad tabell eller vy.
- Skapa ett versionshanterat, helt fiktivt SQL-script (backend/database/sqlserver-preview-poc.sql) för schema och seed-data, utan servernamn, connection string eller produktionsdata. Kör det inte mot projektägarens SQL Server utan uttryckligt godkännande.
- Inför minimal, konfigurationsstyrd lokal CORS-hantering (t.ex. tillåtet origin http://localhost:4200 som exempelvärde), utan wildcard och utan interna URL:er.
- Uppdatera RuntimeConfigService så att en valfri, ignorerad runtime-config.local.json slås samman kontrollerat med den obligatoriska, versionshanterade runtime-config.json (useMockData: true som grund). Sammanslagningen ska bevara övriga feature flags och inte ersätta hela features-objektet. Saknad lokal fil ska ge normalt standardläge, inte ett användarsynligt fel. En trasig lokal fil ska ge ett tydligt, kontrollerat konfigurationsfel. Skapa endast runtime-config.local.example.json som versionshanterad mall.
- Lägg till en previewrad-hämtning i DataCatalogService (mock/API-gren) och utöka DatasetFieldsPreviewComponent med ett valfritt, bakåtkompatibelt stöd för flera riktiga rader från API:et, kopplat in i data-detail.component.ts för dataset-sales-transactions-demo.
- Skapa ett minimalt backendtestprojekt med rena, databasfria enhetstester av registrets allowlist-logik (känt kontra okänt dataset-id).
- Lägg till bin/ och obj/ i .gitignore.
- Skriv minimal svensk dokumentation (t.ex. backend/README.md) för lokal körning, SSMS-steg för det fiktiva scriptet, och en säker user-secrets-mall med platshållare.

## Requirements

- En enda fiktiv datamängd (dataset-sales-transactions-demo) räcker; ingen kataloglistendpoint (GET /api/datasets).
- Ingen PostgreSQL, ingen autentisering, ingen auktorisering, ingen deployment.
- Ingen generell SQL-frågemotor och ingen dynamisk SQL; tabell- och kolumnnamn är alltid hårdkodade literaler från registret, aldrig konstruerade från indata eller klientstyrda parametrar.
- Frontend får aldrig ansluta direkt till SQL Server och får aldrig skicka tabellnamn, kolumnnamn eller SQL-fragment.
- Ett okänt eller otillåtet dataset-id ger ett kontrollerat 404-svar innan någon SQL körs.
- Ingen verklig connection string, inget servernamn, inga credentials eller secrets skapas, skrivs i filer eller skrivs i dokumentation/handoff.
- Backend äger sina egna DTO:er och kontrakt och beror inte på Angulars TypeScript-modeller.
- Ingen ändring av informationsmodellen (Dataset/DatasetFieldPreview) om befintliga objekt räcker.
- Kod och tekniska namn på engelska; dokumentation och UI-text på svenska.
- Ingen commit, push, rebase, merge eller deploy.
- SQL-scriptet får skapas men ska inte köras mot projektägarens lokala SQL Server utan uttryckligt godkännande; verklig databasanslutning och full E2E-verifiering pausas tills projektägaren bekräftat att scriptet är kört och lokal secret är satt.

## Acceptance Criteria

- npm.cmd --prefix frontend start fungerar oförändrat direkt efter klon av main, helt utan SQL Server, backend eller user secrets (mockläge intakt som versionshanterad standard).
- Backend kan startas lokalt och exponerar GET /health, GET /api/datasets/{id} och GET /api/datasets/{id}/preview för dataset-sales-transactions-demo.

- Ett okänt dataset-id mot /preview eller /{id} ger 404 utan att SQL körs.
- Med lokal runtime-config.local.json som aktiverar API-läge visar DatasetFieldsPreviewComponent för dataset-sales-transactions-demo fiktiva rader hämtade från SQL Server istället för den klientderiverade raden, och övriga feature flags bevaras vid sammanslagningen.
- Utan runtime-config.local.json fungerar portalen exakt som idag i mockläge, utan användarsynligt fel.
- En trasig runtime-config.local.json ger ett tydligt, kontrollerat konfigurationsfel, inte en krasch eller tyst felaktigt beteende.
- CORS tillåter endast ett konfigurationsstyrt, exempelvis lokalt origin, ingen wildcard.
- Inga secrets, connection strings, servernamn eller produktionsdata finns i repositoryt.
- npm.cmd run project -- validate, git diff --check, frontend build/test och (efter SDK-verifiering) dotnet build/test är gröna, eller avvikelser är tydligt rapporterade.

## Genomförande (delrapport – väntar på lokal databasverifiering)

Implementerat på branch `docs/AB-027-visa-fiktiv-sqlserver-preview-lokalt-api`:

**Backend** (`backend/`, nytt .NET 10 Web API-projekt, skapat med `dotnet new` och
verifierat med faktisk SDK, inte gissade projektfiler):
- `Portal.slnx` (solution, .NET 10:s nya XML-solutionformat), `Portal.Api/`
  (API-projekt) och `Portal.Api.Tests/` (xUnit-testprojekt).
- `Contracts/DatasetDetailDto.cs`, `DatasetFieldDto.cs`, `DatasetPreviewDto.cs` –
  backendägda DTO:er, inget beroende till Angulars TypeScript-modeller.
- `Datasets/KnownDatasetsRegistry.cs` – enda register för känt dataset-id
  (`dataset-sales-transactions-demo`), statisk metadata och SQL-mappning
  (tabellnamn + kolumnordning). Läses av både controllerns metadata-action och
  adaptern – ingen duplicerad allowlist.
- `Datasets/IDatasetSourceAdapter.cs` / `SqlServerDatasetSourceAdapter.cs` – kör en
  parameteriserad `SELECT TOP (@maxRows) ...` mot ett hårdkodat tabellnamn/kolumner
  från registret. Tar aldrig emot tabellnamn, kolumnnamn eller SQL-fragment från
  anroparen. Ett okänt dataset-id returnerar null (→ 404) innan någon
  SQL-anslutning öppnas.
- `Controllers/DatasetsController.cs` – `GET /api/datasets/{id}` (statisk,
  registerbaserad metadata, kräver ingen SQL Server) och
  `GET /api/datasets/{id}/preview` (riktig SQL-läsning, radbegränsad till 10).
- `Program.cs` – konfigurationsstyrd CORS (`AllowedOrigins`, standard
  `http://localhost:4200`, ingen wildcard), `GET /health`, och en global,
  kontrollerad felhanterare som aldrig exponerar stack trace, connection string
  eller servernamn till klienten (fel loggas fullständigt endast server-side).
- `appsettings.json` – tomt `ConnectionStrings:Default`-platshållarvärde,
  `AllowedOrigins`. `UserSecretsId` satt i `Portal.Api.csproj` för
  `dotnet user-secrets`.
- **Designbugg hittad och fixad under egen verifiering:** `SqlServerDatasetSourceAdapter`
  validerade ursprungligen connection string i sin konstruktor. Eftersom adaptern
  är constructor-injicerad i `DatasetsController` gjorde det att även den SQL-fria
  metadata-endpointen krävde en konfigurerad connection string för att
  controllern överhuvudtaget skulle kunna instansieras. Fixat genom att flytta
  valideringen till `GetPreviewRowsAsync` (lat kontroll, endast vid faktisk
  previewläsning). Verifierat manuellt efter fix: `/health` och
  `/api/datasets/dataset-sales-transactions-demo` svarar 200 utan någon
  connection string konfigurerad; `/preview` ger ett kontrollerat 500-fel utan
  läckta detaljer; okänt id ger 404 på båda endpoints utan att SQL körs.
- `backend/database/sqlserver-preview-poc.sql` – fiktivt, återställningsbart
  script (`DROP TABLE IF EXISTS` + `CREATE TABLE` + fem fiktiva rader) för
  `dbo.SalesTransactionsDemoPreview`. Inget servernamn, ingen connection string,
  ingen databas-`USE`-sats.
- `Portal.Api.Tests/KnownDatasetsRegistryTests.cs` – fyra databasfria enhetstester
  av registrets allowlist-logik (känt/okänt id, fält-/kolumnparitet).
- `backend/README.md` – körinstruktioner, SSMS-steg, user-secrets-mall med
  platshållare, endpoints, återställningssteg.

**Frontend**:
- `frontend/src/app/core/config/runtime-config.service.ts` – läser nu även en
  valfri `runtime-config.local.json`. Saknas filen (normalfallet) används enbart
  grundkonfigurationen, inget fel. En lokal fil som inte går att tolka (ogiltig
  JSON eller fel form på `features`/`systemUrls`) kastar ett tydligt,
  kontrollerat konfigurationsfel istället för att tyst falla tillbaka. Giltig
  lokal override slås ihop kontrollerat – bevarar övriga feature flags och
  systemUrls-nycklar, ersätter inte hela objekt.
- `frontend/public/assets/config/runtime-config.local.example.json` (ny,
  versionshanterad mall) och uppdaterad `frontend/public/assets/config/README.md`.
  `runtime-config.local.json` själv är redan undantagen i `.gitignore` sedan
  tidigare (mönstret fanns, men användes inte förrän nu).
- `frontend/src/app/models/dataset-preview.model.ts` (ny) – liten,
  frontend-ägd spegling av backendens `DatasetPreviewDto`, exporterad via
  `models/index.ts`.
- `frontend/src/app/services/data-catalog.service.ts` – ny metod
  `getDatasetPreview(id)`. I mockläge (standard) härleds exakt samma syntetiska
  rad som tidigare, client-side, från `dataset.sampleFields`. I lokalt API-läge
  anropas `GET {apiBaseUrl}/datasets/{id}/preview` – frontend ansluter aldrig
  direkt till SQL Server.
- `frontend/src/app/shared/components/dataset-fields-preview/` – komponenten har
  nu två nya, valfria inputs (`previewRows`, `previewError`), helt bakåtkompatibla:
  saknas de helt (befintligt beteende, mockläge) visas exakt samma klientderiverade
  rad som innan denna AB. Anges `previewRows` visas riktiga rader (en eller flera).
  Är `previewError` sann visas ett kontrollerat felmeddelande istället för
  tabellen – ingen tyst fallback till den syntetiska raden vid ett misslyckat
  API-anrop, enligt kravet på kontrollerad felhantering.
- `frontend/src/app/features/data-detail/data-detail.component.ts/.html` –
  hämtar previewtillståndet parallellt med övrig relaterad data och skickar
  `previewRows`/`previewError` till komponenten för `dataset-sales-transactions-demo`.
  Övriga fält på sidan (namn, beskrivning, relationer, ägare) kommer fortsatt
  alltid från mock i denna POC, oavsett läge – i linje med den godkända Fas
  1-avgränsningen (endast previewraden bevisar den vertikala kedjan).

**Root**: `.gitignore` – tillagt `bin/`/`obj/` för det nya .NET-projektet.

**Komplettering efter projektägarens första lokala körning:** SQL-scriptet
kördes av misstag första gången med `master` som aktiv databas. Scriptet
(`backend/database/sqlserver-preview-poc.sql`) har därför fått ett generiskt
skydd: om `DB_NAME()` är `master`, `model`, `msdb` eller `tempdb` skriver
scriptet ett tydligt `RAISERROR`-meddelande och sätter `SET NOEXEC ON` (och
återställer `SET NOEXEC OFF` i slutet), så resten av scriptet inte körs mot en
systemdatabas. Skyddet hårdkodar eller antar aldrig projektägarens lokala
databasnamn – det känner enbart igen de fyra fasta systemdatabasnamnen.
`backend/README.md` avsnitt 1 är omskrivet för att göra tydligt att en egen
lokal utvecklingsdatabas måste väljas eller skapas **innan** scriptet körs.
