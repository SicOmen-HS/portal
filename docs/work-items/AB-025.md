## Background

AB-024 införde grundregler för branchbaserat arbete och kopierbara leveranser, men projektets operativa dokumentation behöver samla hela flödet från en komplett task prompt till Worksmith, lokal verifiering, GitHub-review, merge och efterarbete.

Praktiska frontendkommandon behöver kunna köras från repositoryts rot och dokumentationen behöver skilja tydligt mellan agentens interna manifestunderlag och projektägarens normala promptgränssnitt.

## Goal

Dokumentera ett komplett, säkert och ägarstyrt leveransflöde för repositoryagenter och projektägare, inklusive verifierade frontendkommandon från repositoryts rot.

## Scope

- Förtydliga varaktig ansvarsfördelning i docs/project/PROJECT_RULES.md endast där det behövs.
- Samla det fullständiga operativa flödet i docs/project/PROJECT_WORKFLOW.md.
- Förtydliga manifestets tekniska roll i docs/development/PROJECT_ADMINISTRATION.md.
- Dokumentera root-baserad frontendstart i docs/08_Lokal_utvecklingsmiljö.md och praktiska build-/testkommandon i docs/13_Utvecklarguide.md samt bedöm frontend/README.md och docs/09_Teststrategi.md för konsistens.
- Uppdatera docs/project/DOCUMENT_INDEX.md endast om discoverability eller beskrivningar faktiskt påverkas.
- Tillåt kö- och arbetsobjektsändringar som Worksmith själv utför.

## Requirements

- Beskriv en enda komplett task prompt som projektägarens normala gränssnitt och Worksmith-manifestet som tillfälligt internt maskinunderlag med id auto.
- Dokumentera agentens branchansvar och uttryckliga projektägarauktorisation samt stoppvillkor för orelaterat lokalt arbete.
- Dokumentera verifiering före och efter rebase mot origin/main, scoped commit/push, kontroll av fjärrbranch och PowerShell-rutin för GitHubs PR-sida.
- Dokumentera konkreta GitHub-steg för PR, review, merge och branchborttagning utan att införa en ny mergepolicy.
- Dokumentera säker efter-merge-rutin som verifierar innehåll i main utan krav på exakt commit-hash.
- Dokumentera endast scripts som faktiskt finns i frontend/package.json, inklusive start från repositoryts rot, faktisk lokal adress och Ctrl+C.
- Lägg inte till paket, scripts, Worksmith-kod eller applikationsfunktionalitet.

## Verification

- Kör npm.cmd run project -- validate, git diff --check, git status --short --branch och granska git diff.
- Verifiera frontendens startkommando och lokala adress mot package.json, Angular-konfiguration och faktisk startup-utdata.
- Kontrollera relativa Markdown-länkar, avsedda filer och att package.json samt låsfiler är oförändrade.

## Acceptance Criteria

- Dokumentationen beskriver hela flödet från task prompt till efter merge med tydliga auktorisationsgränser och stoppvillkor.
- Manifest, branch, verifiering, fjärrbranch, PR, review, merge och cleanup har entydiga PowerShell-kompatibla instruktioner.
- Frontend kan startas från repositoryts rot med dokumenterat befintligt script och korrekt lokal adress.
- Worksmith-validering och diffkontroller passerar utan nya fel eller varningar.

## Out of Scope

- Ändring av Worksmiths kod, schema eller livscykel.
- GitHub CLI, nya npm-paket, branch protection eller ny mergepolicy.
- Applikationsfunktionalitet, mockdata, commit, push, pull request, merge eller deploy.

## Deliverables

- Uppdaterad styrande och operativ dokumentation utan konkurrerande nytt dokument.
- Verifierat Worksmith-arbetsobjekt i needs_review och en komplett kopierbar handoff.
