## Background

Datamarknad visar idag datamängder, dataprodukter och fältmetadata enbart via lokal, fiktiv mockdata (frontend/public/assets/mock/) laddad genom MockDataService och DataCatalogService. AB-023 lade nyligen till en fältsektion och en syntetisk previewrad på datamängdsdetaljen, uteslutande i mockdatalagret.

Projektägaren vill avgränsa en första teknisk kedja för att senare kunna byta ut mock-JSON mot ett riktigt lokalt API, utan att frontend någonsin får en direkt databasanslutning: Angular -> .NET Web API -> metadataadapter -> lokal SQL Server med helt fiktiv data och metadata. Detta är inte en bred analys av portalens hela framtida backendarkitektur, utan en liten avgränsning av en enda POC-kedja för Datamarknad.

## Goal

Ta fram en avgränsad analysrapport som beskriver minsta möjliga POC för kedjan Angular -> .NET Web API -> metadataadapter -> lokal fiktiv SQL Server för Datamarknadens datamängdsflöde, inklusive minsta API-yta, backendägda API-kontrakt, ett adapterinterface på konceptnivå, en minimal lokal SQL Server-struktur, säker hantering av lokal connection string, och hur Angular-frontenden senare kan växla mellan mock-JSON och API utan att komponenter behöver känna till datakällan.

## Scope

- Analysera nuvarande mockdataflöde för Dataset/datamängder i Datamarknad (modeller, services, mockdata) som grund för avgränsningen.
- Föreslå minsta möjliga API-yta (t.ex. en enda detalj-endpoint kontra lista + detalj) för att bevisa kedjan.
- Föreslå backendägda API-kontrakt/DTO:er som kan mappas till befintliga Angular-modeller utan att importera eller bero på dem.
- Föreslå ett metadataadapterinterface på konceptnivå mellan backendens applikationslager och lokal SQL Server.
- Föreslå en minimal lokal SQL Server-struktur med helt fiktiv data och metadata.
- Föreslå säker hantering av lokal connection string (t.ex. .NET user secrets eller ignorerad lokal konfiguration).
- Analysera hur Angular-frontenden senare kan växla mellan befintlig mock-JSON och ett lokalt API, med minsta möjliga lösning.
- Föreslå små, separata framtida AB-items för implementation av POC:n.
- Dokumentera analysen som en svensk analysrapport under docs/analysis/ och uppdatera detta work item med resultat och rekommenderad uppföljning.

## Genomförande

Analysen dokumenterades i
`docs/analysis/AN-009_lokal_backend_poc_datamarknad.md` efter läsning av det
befintliga mockdataflödet för Dataset (`mock-data.service.ts`,
`data-catalog.service.ts`, `data-detail.component.ts`, `dataset.model.ts`,
`dataset-field-preview.model.ts`, `datasets.mock.json`,
`dataset-fields-preview`-komponenten från AB-023), frontendens
konfigurationsmodell (`runtime-config.model.ts`,
`config/examples/appsettings.example.json`) och relevanta avsnitt av
`docs/04_Systemarkitektur.md` (adapterlistan som redan namnger
`SqlServerAdapter`, samt avgränsningen mellan PostgreSQL som portalens egen
applikationsdatabas och externa datakällor).

**Huvudslutsats:** kedjan Angular → .NET Web API → metadataadapter → lokal
fiktiv SQL Server kan avgränsas till en enda detalj-endpoint
(`GET /api/datasets/{id}`) som första bevisbara steg, med backendägda
DTO:er (`DatasetDetailDto`/`DatasetFieldDto`) som strukturellt speglar men
inte importerar `Dataset`/`DatasetFieldPreview`, ett adapterinterface
(`IDatasetMetadataAdapter`/`SqlServerAdapter`) som isolerar SQL Server-
åtkomst, en minimal SQL-struktur med separata dataset- och fälttabeller,
secrets via .NET user secrets, och en frontend-växling byggd på den redan
existerande `RuntimeConfig.features.useMockData`/`apiBaseUrl`-mekanismen.
Ingen informationsmodelländring krävs. Rapporten (avsnitt "Föreslagna
framtida work items") föreslår fem små, separata AB-item för själva
implementationen.

Primary topic: backend
Secondary topics: data-catalog, integration, configuration, security, frontend
Disposition: accepted_now
Disposition note: Två öppna frågor (listendpoint i första AB:t, adapterns exakta namn) kvarstår för projektägarbeslut innan respektive implementations-AB skapas, se rapportens avsnitt "Öppna frågor".
Follow-up: Etablera minimal .NET Web API-struktur; skapa SqlServerAdapter/metadataadapter och lokal fiktiv SQL Server-datakälla; införa backendägda API-kontrakt och GET /api/datasets/{id}; koppla DataCatalogService/MockDataService till API via RuntimeConfig; verifiera hela lokala kedjan end-to-end (fem separata, framtida AB-item enligt rapportens avsnitt "Föreslagna framtida work items").

den ursprungliga analysen tolkade POC:n som huvudsakligen fokuserad på
hämtning av datasetmetadata, medan huvudmålet även omfattar att praktiskt
bevisa en riktig, begränsad `SELECT` mot en lokal SQL Server-tabell eller
vy med fiktiva datarader.

en ny endpoint (`GET /api/datasets/{id}/preview`), en ny DTO

och en tillåten tabell/vy (server-side allowlist, parameteriserade frågor,
inga klientstyrda tabellnamn eller SQL-fragment), en tredje SQL-tabell/vy
för fiktiva previewrader, samt uppdaterade framtida AB-item, risker och
öppna frågor. Se rapportens nya avsnitt "Komplettering" för fullständig
lista över ändrade avsnitt.
